### 2.3 差距分析

端到端 30% vs 模块 80%——根因：

1. **pipeline 分析阶段依赖关键词触发**：category=code 触发 code 模块分析生成结构化数据 → decisionRouter 匹配规则 → 输出 accelerate/resonate。category=general 时 pipeline 不生成任何结构化数据 → decisionRouter 永远返回 none。

2. **decisionRouter.evaluate() 要求传入对象**：直接传字符串 `t.input` 给 evaluate() 会触发第一行的 `if (!result || typeof result !== 'object')` 检查，直接返回 null。正确用法是通过 dispatch → wrapDispatchResult 传入模块分析结果。

3. **修复方向**：pipeline 增加"默认分析"阶段——即使 category=general，也生成 quality/confidence 等基础字段，让决策路由有默认规则可匹配。

## 3. 逻辑推理能力测试

### 3.1 当前状态：完全空白 (0%)

| 能力 | 状态 | 说明 |
|------|------|------|
| 推理类型检测 | ❌ | 不知道用户需要演绎/归纳/因果/类比推理 |
| 前提检查 | ❌ | 不检查前提是否完整、有无隐含假设 |
| 谬误识别 | ❌ | 不检测循环论证/滑坡谬误/虚假二分等 |
| 推理框架推荐 | ❌ | 不给 LLM 推荐推理方法（三段论/反证法/归纳法） |

### 3.2 测试数据

```
输入: "A > B, B > C, 那么 A 和 C 的关系是？"
  → whatIsThis: type=general cat=general emo=neutral
  → decision: null
  → conclusion: "这是一个待回答的问题。输入包含疑问句。分析问题后给出准确答案。"
  → 心虫没有输出任何推理相关的结构化数据

输入: "所有鸟都会飞。企鹅是鸟。企鹅会飞吗？这个推理错在哪里"
  → 同上，完全没检测到"三段论"或"前提错误"信号

输入: "我该不该辞职去创业"
  → 同上，没检测到"决策推理需求"
```

### 3.3 正确定位

心虫不是推理引擎（不替代 LLM 做推理），是推理辅助层。期望的增量：
1. 检测用户需要什么类型的推理（演绎/归纳/因果/类比）
2. 检查前提是否完整、有无隐含假设
3. 检测逻辑陷阱（循环论证/滑坡谬误/虚假二分/肯定后件等）
4. 给 LLM 推荐推理框架（"建议用三段论分析""建议用反证法"）

当前这 4 项全未实现——这是心虫最核心的空白。

### 5.3 增量模式总结

| 模式 | 表现 | 触发条件 |
|------|------|---------|
| 先承认情绪再解决问题 | 心虫的 emotion 检测让模型先做情绪响应 | anger/sadness/fear 检测准确时 |
| 认知重构 | 为负面体验提供正当性，降低自责 | 愤怒/矛盾场景 |
| 主动推理 | 模型主动推理信息来源或方法论 | memory/decision 类别 |
| 无差异 | 代码分析/逻辑推理/闲聊 | 纯认知场景 |

### 5.4 API 调用方式（代码片段）

```javascript
// 腾讯云 Copilot API
const payload = {
  model: "deepseek-v4-flash",
  messages: [
    { role: "system", content: systemPrompt },
    { role: "user", content: inputText }
  ],
  stream: true,
  temperature: 0.7,
  max_tokens: 2000
};

// SSE 解析
for await (const line of response) {
  if (line.startsWith("data: ") && line !== "data: [DONE]") {
    const chunk = JSON.parse(line.slice(6));
    const content = chunk.choices[0].delta.content || "";
    fullText += content;
  }
}
```
