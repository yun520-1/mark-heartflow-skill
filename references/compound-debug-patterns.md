# Compound Debug Patterns 参考手册
来源: compound-eng-debugging (ClawHub) + HeartFlow 调试引擎整合

## 核心法则：Iron Law

**永远先找根因，再修。** "quick fix now, investigate later" 禁止。
这条规则在时间压力下尤其重要——越是觉得"很明显"的时候，越要查。

**明显 bug 的定义**：错误信息直接说明了原因（如 `ModuleNotFoundError: no module named foo`）。
**不是明显 bug**：错误显示了位置但没说原因（如 `TypeError: Cannot read 'id' of undefined`）。

---

## 六类失败分类

每个 bug 都在以下一个类别中有根因：

| 类别 | 症状 | 典型例子 |
|------|------|----------|
| **逻辑错误** | 输入有效但输出错误 | `≤` vs `<` 循环边界 |
| **数据问题** | null、类型错误、缓存过期 | 上游 API 改了字段名 |
| **状态问题** | 竞态条件、共享状态泄漏 | 单独通过，suite 失败 |
| **集成失败** | 边界契约不匹配 | service A 发 `user_id`，B 要 `userId` |
| **资源耗尽** | 超时、OOM、连接池耗尽 | DB 连接池被打满 |
| **环境问题** | 配置漂移、版本错误 | macOS 通过，Linux 失败 |

## Competing Hypotheses 分析法 (ACH)

当根因不明确时，用 ACH 系统化分析：

### 步骤

1. **列假设**：每个失败类别至少一个假设，越具体越好
2. **收集证据**：对每个假设收集 FOR 和 AGAINST 证据
3. **打分**：>80% 可修，50-80% 继续调查，<50% 禁止修
4. **验证最高分假设**：一次改一个，完全回退如果错了

### 证据强度

| 强度 | 类型 | 例子 |
|------|------|------|
| **强** | 直接观察 | 栈追踪指向具体行 |
| **中等** | 相关性 | bug 出现在 deploy X 之后 |
| **弱** | 口碑 | "我觉得之前见过"（无日志） |
| **可变** | 无证据 | "这个组件没错误"（无≠证明） |

### 置信度阈值

```
>80%: 证据强，反证弱 → 可以修，但要验证
50-80%: 证据混合 → 继续调查再修
<50%: 证据弱或矛盾 → 不要修
```

## 防御纵深验证 (Defense-in-Depth)

修一个 bug 时，不是加一处检查，而是**让 bug 在结构上不可能**。

### 四层验证

1. **入口验证**：拒绝明显无效输入
2. **业务逻辑验证**：即使通过了入口，也要验证领域特定约束
3. **环境守卫**：防止危险操作（如测试环境禁止 git init 非临时目录）
4. **调试插桩**：当前几层都失败时，捕获取证上下文

### 应用

修 bug 时：
1. 追踪数据流——坏值从哪来？被谁消费？
2. 映射所有检查点——列出数据经过的每个函数/边界
3. 每层加验证
4. 独立测试每层

---

## 三次修复阈值

**3 次失败修复尝试后，停止。** 一次尝试 = 一个完整的"假设→验证"循环。

架构问题信号：
- 每个修复都暴露了新的共享状态或意想不到的耦合
- 修复需要大规模重构才能正确实现
- 每个修复在其他地方产生新症状

→ 升级给用户，说明已尝试的方法和架构顾虑。

## 根因追溯 (Root Cause Tracing)

错误出现在调用栈深处时：
- **不要在错误出现的地方修**
- **从症状向后追溯调用链**，找到原始触发点，在源头修

### 追溯过程

1. 观察症状（错误信息）
2. 找到直接原因（哪行代码触发的）
3. 向上追溯调用者
4. 追踪坏值（在每一层问：传了什么值？从哪来的？）
5. 找到原始触发点

### 栈追踪插桩

```javascript
async function dangerousOperation(arg) {
  console.error('DEBUG dangerousOperation:', {
    arg,
    cwd: process.cwd(),
    stack: new Error().stack,
  });
  // ... proceed
}
```
用 `console.error()`（不是 logger，测试中可能被抑制）。在操作**前**记录，不在失败后。

## 调试报告模板

每个 resolved bug 输出结构化报告：

```
SYMPTOM:    [观察到什么]
ROOT CAUSE: [为什么发生 — file:line + 证据，深度≥2层]
FIX:        [改了什么]
EVIDENCE:   [证明修复的验证输出]
REGRESSION: [加了什么测试防止复发]
RELATED:    [同类历史 bug、已知问题、架构笔记]
STATUS:     DONE | DONE_WITH_CONCERNS | BLOCKED | NEEDS_CONTEXT
```

## 红旗（停止并回到步骤1）

| 你在做什么/想什么 | 真正意味着什么 |
|-------------------|----------------|
| "shotgun debugging" / "我看出问题了" | 推理不是证据。先形成假设，再改，再测。 |
| 忽略间歇性失败 | 在负载下插桩和重现。隔离成功不能解释集成失败。 |
| "之后清理调试代码" | 现在删，否则会 shipped 到生产。 |
| "这个失败是预先存在的，和我们改动无关" | 证明它：在 base 分支跑测试。 |
| "测试错了，不是代码" | 验证后再否定。读测试的意图。 |
| "参考太长了，我按自己理解改" | 局部理解保证 bug。完整读完 working 示例再应用。 |

## 间歇性问题信号

- **竞态条件**：共享可变状态，check-then-act 模式，缺少锁。Node.js async 中：`.then()` 链交错，协程间无守卫共享状态。
- **死锁**：循环锁获取（在多查询间持有 DB 行锁），async 代码中循环 await 依赖。
- **资源耗尽**：连接池耗尽时检查 pool size vs 并发请求数，验证错误路径上连接是否被归还。
- **时序依赖**：用条件轮询代替 `sleep()` —— 等待实际状态，不等固定时长。

## CI 失败分类

1. **fetch logs**: `gh run view <run_id> --log`
2. **分类**: build error / test failure / lint-type error / timeout / infrastructure
3. **本地重现**: 跑 CI 配置里的相同命令
4. **修复和验证**: 修复后重跑检查

不要无改变地重试 CI run。两次相同失败 = broken，不是 flaky。
