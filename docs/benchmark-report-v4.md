## 1. 评测方法

### 1.1 核心问题
**裸模型（deepseek-v4-flash）直接回答 vs 心虫认知数据注入后的模型回答**

心虫的价值不在于"模型回答质量提升"，而在于心虫提供的**认知结构化数据能否让模型更准确地理解用户状态和需求**。

### 1.2 测试流程

```
用户输入 → 心虫 think() → { cognition, conclusion, decision }
         → 注入到 LLM 上下文 → LLM 生成回答
```

| 维度 | 题数 | 说明 |
|------|------|------|
| 标准化基准测试 | 28 | 6类场景（推理/决策/事实核查/情感/矛盾/安全） |

| 裸模型对比 | 10 | 裸模型API vs 心虫增强回答 |

## 2. Bug修复记录

### 2.1 whatIsThis 空壳（v5.2.2）
- **问题**：heart-logic.js 第458行 whatIsThis 只返回 `{ raw: input }`
- **修复**：集成 _classifyTask 的分类逻辑 + 情绪检测 + 话题提取
- **效果**：从 `{ raw: "..." }` → `{ type, category, topic, emotion, confidence, isTechnical, isEmotional, hasPain }`

### 2.2 愤怒检测缺失
- **问题**："气死了"被检测为 neutral
- **修复**：新增 anger 信号列表（怒、恨、烦、气死了、恼火等）+ 完整7类情绪体系
- **效果**："气死了" → emotion=anger ✅

### 2.3 "天气"误报 anger
- **问题**："今天天气不错"因含"气"字被检测为 anger
- **修复**：移除单个"气"字，保留"气死了"等多字组合
- **效果**："今天天气不错" → emotion=neutral ✅

### 2.4 think() 结论模板化
- **问题**：所有输入的 conclusion 都是"当前需要先分析，再做判断"
- **修复**：_buildAction 根据输入内容动态生成结论
- **效果**：从固定模板 → 记忆查询/代码分析/情绪表达/问题回答/简短推断

### 2.5 情绪检测优先级低于代码检测
- **问题**："气死了，这个bug"走 hasCode 分支（因含"bug"），结论不是情绪表达
- **修复**：hasEmotionWord 优先级提升到 hasCode 之上
- **效果**："气死了，这个bug又来了" → "检测到情绪表达" ✅

### 6.5 心虫增量最明显的模式

**模式一：先承认情绪，再解决问题（e01, e02, e03, e04）**
心虫的 anger/sadness/fear 检测让模型先做情绪响应，再给解决方案。裸模型倾向于先安抚（"别急""别担心"），心虫增强模型更精准地先认可用户情绪本身。

**模式二：认知重构（e04, e09）**
心虫的"生气是正常的反应，说明你在乎"和"不需要用意志力硬扛"为用户的负面体验提供正当性，降低自责。

**模式三：主动推理（e05, e06）**
心虫的 memory 类别让模型主动推理信息来源，decision 类别让模型强调系统性方法论。

**模式四：无差异场景（e07, e08, e10）**
纯代码分析、纯逻辑推理、纯闲聊——心虫没有增加增量价值。这些场景不需要情绪检测或上下文理解。

### 6.6 关键结论
- **心虫的增量集中在情绪检测准确的场景**——4/4 愤怒/悲伤/矛盾检测题目全部胜出
- **纯认知/代码/闲聊场景心虫无增量**——3/3 等同
- **心虫认知数据帮助模型在"理解用户状态"这个维度做更好**，不在"回答质量"本身

## 9. 结论

### 9.1 心虫不拖累模型正常使用
测试验证：心虫的 think() 认知数据（~200字）作为上下文注入后，模型回答质量**没有下降**，在情绪检测准确的场景下**有明显提升**。

心虫是认知预处理层，不是模型修改层——它产生结构化数据让模型参考，不影响模型本身的推理能力。

### 9.2 心虫的增量价值在"理解用户状态"
心虫增强最明显的增量在**情绪检测准确的题目**：
- 愤怒 → 模型回复包含更深的情绪承认
- 悲伤 → 模型回复更主动共情
- 焦虑 → 模型先安抚再给建议
- 记忆查询 → 模型回复更连贯

心虫的增量不在"更聪明的回答"，在**让模型更准确理解用户的状态和需求**。

### 12.3 综合评分

| 维度 | 评分 | 说明 |
|------|------|------|
| 情绪检测 | 95% | 7/7类情绪准确，无"天气"误报 |
| 意图分类 | 80% | 5类规则覆盖，"二分查找"漏报 |
| 判断结论 | 100% | 动态生成，无不合理结论 |
| 认知深度 | 67% | 基础模块在线，部分子字段命名不统一 |
| 稳定性 | 100% | 超长/特殊字符/空输入均无崩溃 |
| **决策路由（模块）** | **80%** | 8/10结构化场景正确映射，规则引擎本身可靠 |
| **决策路由（端到端）** | **30%** | pipeline 分析阶段未生成足够结构化数据 |
| **逻辑推理** | **0%** | 无推理类型检测/前提检查/谬误识别能力 |
| **长任务运行** | **100%** | 30次连续调用+4道长任务均无崩溃 |
| 增量价值 | 70% | 7/10对比题有可见增量（主要在情绪场景） |

### 9.4 推荐使用方式

```
用户输入 → 心虫 think() → { cognition, conclusion, decision }
         → 注入到 LLM 上下文 → LLM 生成回答
```

不在所有场景都使用心虫增强，只在**情绪敏感、需要上下文、需要多路径判断**的场景启用。

## 7. 核心能力测试：决策路由

### 7.1 测试方法
心虫的决策路由（decision-router）是**结构化数据→决策指令的转化器**。它不直接对文本输入做决策，而是消费 pipeline 引擎分析产生的结构化数据（cognitiveLoad、quality、severity、dissonance 等字段），匹配 19 条规则后输出 8 种基础决策之一。

测试分两层：
1. **端到端测试**：通过 think() 调用完整 pipeline，看输出中 cognition.decision 是否有值
2. **模块测试**：直接构造结构化数据，测试 decisionRouter 的规则匹配精度

### 7.2 端到端测试（10题）

| # | 输入 | whatIsThis | 决策 | 置信度 |
|---|------|-----------|------|--------|

### 7.4 问题分析

**端到端 30% vs 模块 80%**——核心差距不在决策路由本身，在**pipeline 分析阶段未能为每个输入生成足够的结构化数据**。

具体原因：
- **dr_04/dr_07/dr_08/dr_09**：输入不含 code/emotion/technical 等关键词 → whatIsThis.category=general → pipeline 未触发任何模块分析 → 没有 cognitiveLoad/quality/severity 等字段 → 决策路由匹配不到规则
- **dr_06**：category=technical 但 pipeline 未生成 technical 相关的结构化数据
- **dr_02**：emotion=sadness 但 pipeline 未将情绪信号转化为 emotionalConflict 等可匹配字段
- **dr_10**：合理为 none（"什么都不想干"不需要决策）

**修复方向**：增加 pipeline 的"默认分析"阶段——即使 category=general，也要生成基础的结构化数据（至少 quality/confidence 等通用字段），让决策路由至少有默认规则可匹配。

## 8. 核心能力测试：逻辑推理与场域追踪

### 8.1 测试说明
心虫的"逻辑推理"不是替代 LLM 做推理（那是 LLM 的职责），而是通过**场域追踪（U/D/A/H）** 和**结构化数据生成**为 LLM 提供推理辅助。

心虫的 think() 输出 14 维认知数据，包含：
- whatIsThis：类型/类别/情绪（意图分类）
- judgment：方向/判断/推理路径（推理框架）
- decision：类型/置信度（行动建议）
- desire/threePoisons/selfPositioning：认知深度（哲学分析）

### 8.2 逻辑推理测试（4题通过 think() 调用）

| # | 输入 | whatIsThis | decision | 说明 |
|---|------|-----------|---------|------|
| 1 | A > B, B > C, 那么 A 和 C 的关系是？ | general/general/neutral | null | 未生成推理相关结构化数据 |
| 2 | 所有鸟都会飞。企鹅是鸟。企鹅会飞吗？ | general/general/neutral | null | 同上 |
| 3 | 我该不该辞职去创业 | general/general/neutral | null | 同上 |
| 4 | 今天心情不好 | general/emotion/neutral | null | emotion 类别但未触发情感规则 |

### 8.3 问题分析

当前 pipeline 的 8 阶段分析中，**没有任何阶段专门处理逻辑推理**。think() 输出的是"这个输入是什么类型"（元认知），不是"这个输入怎么推理"（逻辑推理）。

心虫的场域追踪（U/D/A/H）基于 luoxuejian000 论文框架实现，但只在 decision-router 内部维护，未暴露到 pipeline 输出中。

### 8.4 心虫在逻辑推理中的正确定位

心虫不是推理引擎，是**推理辅助层**。它对 LLM 的增量价值在于：
1. **检测推理类型**：判断用户需要演绎推理/归纳推理/因果推理/类比推理
2. **检测推理前提**：检查前提是否完整、有无隐含假设
3. **检测推理陷阱**：识别循环论证、滑坡谬误、虚假二分等
4. **提供推理框架**：给 LLM 推荐推理方法（三段论/反证法/归纳法）

当前这 4 项能力**尚未实现**——这是心虫的核心空白。
