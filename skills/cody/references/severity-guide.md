# 严格程度说明

## 🔵 信息模式（Info）

**触发词**：快速 review / 简单看看 / 信息模式

**检查范围**：
- 命名规范（变量名、函数名是否清晰表意）
- 注释完整性（公共函数是否有文档注释）
- 代码格式（缩进、空格、换行）
- 未使用的导入 / 变量

**跳过**：所有功能性问题、安全问题、性能问题

**适用场景**：快速浏览代码风格，不深入逻辑

---

## 🟢 优化模式（Optimize）

**触发词**：优化模式 / 只看优化点

**检查范围**：
- 🔵 信息模式的所有内容
- 可读性问题（过长函数、魔法数字、晦涩逻辑）
- 明显违反最佳实践的地方
- 代码复用（是否有重复逻辑可以抽取）

**跳过**：安全问题、严重 Bug、性能问题

**适用场景**：关注代码质量，但不涉及安全和性能

---

## 🟡 标准模式（Standard）⭐ 默认

**触发词**：review / 代码审查 / 检查代码（未指定模式时）

**检查范围**：
- 🟢 优化模式的所有内容
- 🔴 严重 Bug 和安全漏洞
- 性能问题（明显的 N+1、不必要的循环等）
- 错误处理问题（异常未捕获、返回值未校验）
- 类型安全问题（类型注解缺失、隐式转换）

**跳过**：细微命名偏好、注释风格

**适用场景**：日常代码 review，平衡效率和深度

---

## 🔴 严格模式（Strict）

**触发词**：严格模式 / 严格 review / 全面检查 / PR review

**检查范围**：全部维度，包括：
- 🟡 标准模式的所有内容
- 命名规范（严格检查每个变量名是否表意清晰）
- 注释完整性（所有公共函数必须有文档注释）
- 测试覆盖（关键逻辑是否有对应测试）
- 边界条件处理（所有边界情况是否覆盖）
- 代码复用（严格检查 DRY 原则）
- 错误处理一致性（所有错误处理方式是否统一）

**适用场景**：正式 PR review、关键模块上线前检查

## 模式对比表

|---|---|---|---|---|
| 命名规范 | ✅ | ✅ | ✅ | ✅✅ |
| 注释完整性 | ✅ | ✅ | ✅ | ✅✅ |
| 可读性 | - | ✅ | ✅ | ✅ |
| 最佳实践 | - | ✅ | ✅ | ✅ |
| 性能问题 | - | - | ✅ | ✅ |
| 安全问题 | - | - | ✅ | ✅✅ |
| 严重 Bug | - | - | ✅ | ✅✅ |
| 错误处理 | - | - | ✅ | ✅ |
| 类型安全 | - | - | ✅ | ✅ |
| 测试覆盖 | - | - | - | ✅ |
| 边界条件 | - | - | - | ✅ |

## CI 集成建议

| 场景 | 推荐模式 | CI 门禁 |
|---|---|---|
| Pre-commit hook | 🔵 信息 | 无严重问题即可 |
| PR 自动检查 | 🟡 标准 | 无 🔴 严重问题 |
| 主分支合并 | 🔴 严重 | 无 🔴🟡 问题 |
| 日常开发 | 🟡 标准 | - |
