# Review 维度详解

## 🐛 潜在 Bug

重点检查：
- 空指针 / undefined / None 访问（未判空直接使用）
- 数组越界、off-by-one 错误
- 异常未捕获或捕获后被吞掉
- 条件判断逻辑错误（== vs ===、边界条件）
- 并发问题（竞态条件、非线程安全操作）
- 资源未释放（文件句柄、数据库连接、锁）
- 类型错误（隐式转换、错误的类型断言）
- 返回值未校验（函数返回 None/undefined 但调用方未处理）

## 🔒 安全问题

重点检查：
- SQL 注入（字符串拼接 SQL，未使用参数化查询）
- XSS（未转义的用户输入直接插入 HTML）
- 硬编码密钥 / 密码 / Token / API Key（应使用环境变量）
- 路径遍历（用户输入直接拼接文件路径）
- 不安全的反序列化（pickle、eval、exec）
- 敏感信息日志输出（密码、token 被记录）
- 权限校验缺失（API 未鉴权）
- 依赖已知漏洞版本（如能识别）
- SSRF（用户输入直接用于请求 URL）
- 命令注入（用户输入拼接 shell 命令）

## ⚡ 性能问题

重点检查：
- N+1 查询（循环内执行数据库查询）
- 不必要的全量加载（应分页或按需加载）
- 重复计算（循环内重复调用昂贵操作）
- 低效数据结构（用数组做频繁查找，应用 Map/Set）
- 同步阻塞（可以异步的操作用了同步方式）
- 内存泄漏（事件监听未移除、缓存无上限增长）
- 过早优化（不必要的缓存、复杂度增加但收益小）
- 大对象复制（应使用引用或浅拷贝）

## 📖 可读性

重点检查：
- 函数过长（超过 50 行建议拆分）
- 嵌套过深（超过 3 层建议重构）
- 魔法数字 / 魔法字符串（应定义为常量）
- 晦涩的变量名（`tmp`、`data`、`x`、`val` 等）
- 复杂的布尔表达式（应提取为命名变量）
- 注释与代码不一致（过时注释）
- 缺少必要注释（复杂逻辑无说明）
- 硬编码路径（应使用配置）

## ✅ 最佳实践

重点检查：
- 重复代码（DRY 原则，相同逻辑出现 2+ 次）
- 函数职责不单一（一个函数做了太多事）
- 错误处理不一致（有些地方处理，有些地方忽略）
- 不必要的全局变量
- 过度工程（过度抽象、不必要的设计模式）
- 语言特性误用（如 Python 中用 + 拼接字符串代替 join）
- 未使用的导入 / 变量
- 可简化的逻辑（能用三元表达式却用了 if-else）

## 🧪 类型安全（新增）

重点检查：
- 缺少类型注解（Python type hints、TS 类型）
- 隐式类型转换（可能产生意外结果）
- Any 类型滥用（失去类型检查意义）
- 类型断言不安全（as / cast 未做校验）
- 泛型使用不当
- 可空类型未处理（Optional / Maybe 未解包）

## 🛡️ 错误处理（新增）

重点检查：
- 裸异常捕获（`except:` / `catch (e)` 吞掉所有异常）
- 异常捕获范围过大（应捕获具体异常类型）
- 异常处理后未恢复状态（资源未释放、锁未释放）
- 错误码与异常混用不一致
- 关键操作无重试机制（网络请求、IO 操作）
- 错误信息不明确（用户无法理解问题原因）

## 🧪 测试覆盖（新增）

重点检查：
- 复杂逻辑无单元测试
- 边界条件未覆盖
- 异常路径未测试
- Mock 使用不当（过度 mock 或 mock 真实依赖）
- 测试断言不充分（只验证返回值，未验证副作用）
- 集成测试缺少（只有单元测试，无端到端验证）

---

## 严重程度分级

| 级别 | 含义 | 示例 |
|---|---|---|
| 🔴 严重 | 必须立即修复，会导致安全问题或崩溃 | SQL 注入、空指针、硬编码密钥 |
| 🟡 建议 | 应该修复，影响性能或可维护性 | N+1 查询、过长函数、重复代码 |
| 🟢 优化 | 可以改进，提升代码质量 | 命名优化、添加注释、简化逻辑 |
| 🔵 信息 | 仅供参考，不影响功能 | 代码风格、格式建议 |
