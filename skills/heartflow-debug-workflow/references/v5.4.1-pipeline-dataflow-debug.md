# v5.4.1 Pipeline 数据流调试实录

> 日期：2026-06-28
> 版本：v5.3.0 → v5.4.0 → v5.4.1
> 根因：pipeline decision 阶段只传了4个字段，27条决策规则大部分匹配不到默认值0

## 问题链

```
pipeline decision 阶段只传 inputText/cognitiveLoad/directionClear/confidence/dissonance
  → cognitiveLoad 始终 0（agentPsychology 返回默认值）
  → quality/stability/severity/identityCoherence 全是 undefined
  → 27条规则全部不匹配（各自要求 >0.3 或 <0.4）
  → evaluate() 返回 { decision: null, matched: false }
  → think() 的 cognition.decision.type = null
```

## 修复步骤

### 1. 注入9个字段

在 pipeline.js 的 decision 阶段（约第189行），扩展 inject 对象：

```javascript
// 之前（4个字段）
const fieldData = fieldInjector.inject({
  inputText: ctx.input,
  cognitiveLoad: ctx.psychology?.cognitiveLoad?.load || 0,
  directionClear: 0.3,
  confidence: ctx.judgment?.confidence || 0.5,
  dissonance: 0,
}, 'think') || { inputText: ctx.input };

// 之后（9个字段）
const fieldData = fieldInjector.inject({
  inputText: ctx.input,
  cognitiveLoad: /* 多源汇聚 */,
  directionClear: ctx.judgment?.direction === 'act' ? 0.8 : 0.6,
  confidence: ctx.judgment?.confidence || 0.5,
  quality: ctx.judgment?.confidence ? Math.min(0.9, ctx.judgment.confidence / 10) : undefined,
  stability: ctx.judgment?.stability || undefined,
  severity: ctx.psychology?.pain?.painLevel || undefined,
  dissonance: ctx.psychology?.cognitiveDissonance || 0,
  identityCoherence: ctx.heartLogic?.isRightAction ? 0.9 : 0.5,
  desireDominant: null,
  poisonLevel: null,
}, 'think');
```

**注意**：`fieldInjector.inject()` 只返回它知道的字段。不在 `inject()` 定义中的字段不会出现在返回对象中。如果 `fieldInjector` 未定义，直接返回对象。

### 2. 多源汇聚 cognitiveLoad

cognitiveLoad 从 agentPsych 取到的永远是0（因为 fullAssessment 是静态方法）。改为从输入复杂度计算：

```javascript
const cognitiveLoad = ctx.psychology?.cognitiveLoad?.load ||
                      (ctx.input?.length > 200 ? 0.7 :
                       ctx.input?.length > 100 ? 0.5 :
                       ctx.input?.length > 50 ? 0.3 : 0);
```

### 3. 加兜底规则

在 decision-router.js 的 evaluate() 无匹配分支：

```javascript
if (matches.length === 0) {
  return {
    decision: {
      type: DECISION.HOLD,
      confidence: 0.3,
      priority: 30,
      rationale: '无匹配规则，等待更多数据',
      ruleId: 'default-hold',
      timestamp: Date.now(),
      source,
      fallback: null,
    },
    matched: false,
    rules: [],
    field: fieldData,
  };
}
```

## 关键教训

1. **pipeline 是数据流的核心瓶颈** — 所有模块的数据必须通过 pipeline 的 ctx 对象传递到 decision 阶段。任何阶段返回的数据没被后续阶段使用，就相当于没产生。
2. **不要假设数据会自己流过去** — 每个阶段必须显式从 ctx 读取、转换、传递。不加显式传递逻辑，字段就是 undefined。
3. **兜底规则比 null 好** — 无数据时输出 hold(0.3) 让上层知道引擎在工作但没足够信息。输出 null 会让上层无法处理。
4. **MCP 重启后才能看到效果** — 修改 engine 代码后必须重启 MCP server（launchctl stop/start），否则旧进程继续返回旧结果。
