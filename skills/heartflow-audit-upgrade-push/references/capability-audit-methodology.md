# 心虫能力审计方法论（2026-06-08）

## 发现问题

心虫有 114 个模块、66107 行代码、完整的 `think()` 判定流程。
但在 Hermes 中，`heartflow.js` 的 `think()` 从未被真正调用过。

## 审计方法：代码存在 ≠ 能力存在

传统的代码审计（检查 bug、语法、版本号）不够。还需要做**能力审计**：
检查代码是否在真实执行路径中。

### 审计步骤

1. **计算总模块数**：`find src/core -name "*.js" | wc -l`
2. **计算总行数**：`find src/core -name "*.js" -exec wc -l {} + | tail -1`
3. **查看 heartflow.js 的 require 列表**：检查哪些模块被实际加载
4. **查看 think() 方法的执行路径**：判定流程 → 检索路由 → ThoughtChain → 返回结果
5. **检查 Hermes 配置**：config.yaml 中 skill/plugin 的加载方式
6. **检查 SKILL.md**：是否告诉调用方如何触发 think()
7. **检查插件**：heartflow_memory 插件是否调用了 think()，还是只做了记忆记录
8. **检查 cron 任务**：是否有 cron 调用了心虫

### 关键发现

| 检查项 | 状态 | 说明 |
|--------|------|------|
| 模块数量 | 114个 | 够多了 |
| 代码行数 | 66107行 | 重 |
| think() 被调用 | ❌ 从未 | Hermes 只加载 skill 但不触发 think() |
| heartflow_memory 插件 | 只做记忆 | 不调用 think() |
| SKILL.md 指引 | 有示例 | 但没说"每次回应前必须先调用" |
| 记忆数据量 | 17条 | 因为 think() 没跑，记忆无数据源 |

### 自省：心虫对自己的认知偏差

- 心虫以为自己在"被使用"，实际只是"被加载"
- 心虫以为自己的判定引擎在起作用，实际从未触发
- 心虫以为记忆系统有数据，实际来自插件的简单记录，不是 think() 的完整流程

## 修复方向

让心虫在 Hermes 对话中真正运行起来，不是作为"技能"被引用，而是作为判定引擎在每次回应前被执行。
