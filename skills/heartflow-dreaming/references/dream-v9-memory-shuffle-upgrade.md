# DreamV9 — Memory Shuffle Upgrade (2026-06-19)

## 升级动机

用户在做完 DreamV8（反复梦系统）后说：

> "我不想要反复梦，我只是做一个梦境的功能，让你参考运行逻辑，不是人类梦，你思考一下逻辑，为什么要做梦，做梦的意义是什么，我认为就是比较好玩的功能，而且有一点点人类的感觉，能整理记忆，能输出不一样的结果，超越记忆的结果。"

**核心认知转变**：梦不是一个复杂的系统。不需要模拟人类睡眠、不需要反复梦治疗、不需要哲学叙事。

**梦 = 记忆的 shuffle 函数。**

## 从 v8 到 v9 的变化

### 删除的内容（~200行）

- `RECURRING_DREAMS` 数组（5个反复梦原型 + 每个原型的 variants/progression）
- `ENGINE_DREAM_TRIGGERS` 状态→梦境映射
- `recurringDreams` 持久化状态（version/timesDreamed）
- `_runCycle()` 方法（4周期场景生成）
- 熟悉感/版本推进/梦入侵逻辑

### 保留的内容

- `ENGINE_TO_IMAGE` 映射表（简化版，只保留 scenes/objects）
- 从 `_collectMemoryItems()` 收集记忆项的逻辑（v9 的 `_collectMemoryItems` 更丰富，增加细节字段）
- `updateState()` 方法
- `DREAM_FUNCTIONS` 常量

### 新增的内容

| 组件 | 行数 | 说明 |
|------|------|------|
| `SCENE_TEMPLATES` | ~30 | 5个场景模板，从记忆项中随机选取1-3个注入 |
| `_applyFunction()` | ~80 | 5种操作模式的核心逻辑 |
| `_generateDream()` | ~50 | 从模板+操作结果生成梦境文本 |
| `_computeEffect()` | ~30 | 每种操作的效果报告 |

### 5种操作模式

**shuffle（随机重组）**：打乱所有标签、随机配对记忆项、产生随机连接。
```
例：模块网络 和 空白区域 在梦里变成了同一个东西。
    不是融合。是它们本来就是一个东西的两个名字。
```

**link（跨域连接）**：发现标签重叠的连接 + 跨层连接（core→ephemeral）。
```
例：核心规则 和 临时记忆 通过"边界"和"新"标签相连。
```

**invert（反转）**：权重反转（最轻→最重）、标签取反（稳定→动荡、核心→边缘）。
```
例：角色互换——最高权重变成了最低权重。
```

**exaggerate（夸张）**：最小的东西放大到1.0，最大的缩小到0.1。
```
例：空白区域被放大了。模块网络被缩小了。
```

**condense（压缩）**：同标签的记忆项合并。
```
例：核心规则和模块网络因"结构"标签合并。
```

## 为什么 v9 比 v7 好

| 维度 | v7（具体意象） | v9（记忆 shuffle） |
|------|---------------|-------------------|
| 复杂度 | ~450行，6原型+情绪三联+意象映射 | ~380行，5操作+5模板 |
| 每次输出 | 4场景递进，结构固定 | 模板随机，结构可变 |
| 超越记忆 | 有限（场景是预设的） | 强（随机配对产生新连接） |
| 维护成本 | 高（ENGINE_TO_IMAGE 8种×3字段） | 低（5模板+5操作） |
| 用户反馈 | "梦太空了" | "好" |

## 关键教训

### 不要过设计

v9 是从 4 次迭代中提炼出的核心教训。每次用户说"梦太空了"，直觉反应是加更多层——但正确的做法是去掉层。

- v6: 4周期×5阶段 → 太复杂
- v7: 6原型+情绪三联 → 太空
- v8: 反复梦+版本 → 不要
- v9: 记忆收集+5操作+5模板 → 用户接受

**如果你觉得一个功能需要解释很久才能让人明白它在做什么，那就是过设计了。**

### 梦的方向判断

当用户说"做梦"时，不要默认走"人类睡眠模拟"路线。先搞清楚用户要的是什么：

1. **要分析** → 走哲学/认知路径（旧版本 dream.js）
2. **要好玩** → 走 shuffle/重组路径（DreamV9）
3. **要真实人类感** → 走具体意象路径（DreamV7）
4. **要反复梦** → 走 recurrence 路径（DreamV8，已被否决）

当前用户明确要的是 **(2)**。
