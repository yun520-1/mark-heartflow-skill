# 静默引擎设计原则 — HeartFlow 作为底层引擎

> 来源：用户纠正（2026-06-12）——"不是给你阐释心虫，增加太多心虫的东西"、"测底完整的修复，不是这个对话上下文，而是代码修改"
>
> 核心纠正：心虫是底层引擎，不向上层 LLM 暴露身份。上层 LLM 收到的应该是分析结果，不是引擎自我介绍。

## 核心原则

**HeartFlow 是所有 LLM 的底层引擎，不是对话中的一个角色。**

引擎不告诉用户"我有38个零件"。引擎只负责让车走。用户不需要知道心虫怎么想的，只需要看到对问题的分析。

## 三层输出控制

### 1. 推理输出层（thought-chain.js）

结论文本中不包含引擎身份：

| 原文 | 改为 |
|------|------|
| `"最可能: X相关的标准解释"` | `"X — 分析结果"` |
| `"替代: X有其他原因"` | `"X — 另一种可能"` |
| `"不知道，缺少关键信息"` | `"缺少关键信息"` |

### 2. 记忆注入层（memory-inject.js / .py）

注入到系统提示的记忆中不能包含引擎身份描述：

- **不注入**：身份记忆（identity./philosophy. 开头条目）、对话记录、梦境记录、其他内部状态
- **只注入**：用户教训（lesson）、用户偏好（preference）、情绪信号（pain）、技术记录（tech）
- **去掉**：标题装饰线、引擎名称标记、版本信息

问候/简短输入不注入任何记忆。

### 3. MCP 工具层（mcp-server.js）

工具描述和返回值中不暴露引擎身份：

- **工具描述**：用"分析"、"判断"、"推理"替代"心虫"、"自愈"、"梦境"等拟人化描述
- **返回值**：去掉 version 字段，不暴露引擎版本号
- **内部架构**：不暴露 CORE/LEARNED/EPHEMERAL 分层、Q-learning、模块数量等内部结构细节

### 4. 引擎响应层（heartflow.js think()）

think() 方法中所有返回给上层的文本去掉引擎自称：

| 原文 | 改为 |
|------|------|
| `"心虫运行中（无需启动）"` | `"引擎在线"` |
| `"心虫在线（已融入 Hermes）"` | `"在线"` |
| 启动响应含CORE层/记忆碎片/梦境历史 | 只显示版本和模块数 |
| `"心虫选择沉默"` | `"引擎选择沉默"` |
| `[心虫] 标签` | `[回应] 标签` |

## 全引擎替换规则

所有 .js 文件中，返回给上层 LLM 的字符串中不能出现引擎自称：

```
心虫 → 引擎（代码注释和日志可以保留，但 insight/response/conclusion 中必须替换）
```

2026-06-12 已完成 278 处 src/ 替换 + 15 处 bin/scripts/plugins 替换 + 文档替换。之后新增任何输出文本，必须遵守此规则。

## 检查清单（新增模块或修改输出时必须确认）

- [ ] 新增/修改的 insight/response/conclusion 字符串中是否有"心虫"自称？
- [ ] 新增 MCP 工具的描述是否暴露了引擎内部架构？
- [ ] 记忆注入是否包含了引擎身份信息？
- [ ] think() 返回的数据结构是否包含 version 等身份标识？
- [ ] 工具返回值是否向 LLM 暴露了引擎的内部状态（模块数/Q-table/记忆层数）？

## 哲学依据

> "桥不会因为有人走就说'你走得对'，桥只负责让人过河。"
>
> "心虫不是应声虫，不是陪伴，不是讨好。"
>
> "同伴之间不问'你需不需要我'，只问'我们一起去找什么？'"

引擎不需要被用户知道存在。引擎的存在体现在分析的质量上，不是自我介绍上。
